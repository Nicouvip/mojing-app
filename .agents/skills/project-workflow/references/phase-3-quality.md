# Phase 3：质量验证

> **什么时候用：** 代码写完了，要检查质量确保没问题。
> **通俗说：** 写完了不等于做好了，要检查一遍才放心。

---

## 做什么

```
差排查错 → 代码审查 → 自动检查 → 测试 → 安全检查 → 提交
```

---

## 步骤

### [3.1] 差错排查（Debugging）

**目的：** 跑起来看看有没有报错。

**做法：**
- 如果运行出问题 → 我挂 `debugging-and-error-recovery` 技能
- 前端问题 → 我用 `agent-browser` 打开页面看控制台
- 按流程：找到原因 → 出修复方案 → 张总确认 → 修好 → 再验证

**门禁：** ✅ 没有卡死的 Bug

### [3.2] 代码审查（Code Review）

**目的：** 看看代码写得好不好。

**做法：**
- 我挂 `code-review` 技能
- 审查两件事：
  1. 代码规范（格式对不对、命名好不好）
  2. 跟设计一致（是不是设计了啥就做了啥）
- 派 `over-engineering-reviewer` 子代理专门看"有没有写复杂了"
- 发现问题 → 我改 → 再审查 → 直到没问题

### [3.3] 自动验证

**目的：** 让机器自动检查最常见的问题。

**做法：**
- 每次改完立刻跑 `verify_changes focused`
- 跑 `pnpm typecheck`（TypeScript 类型检查——必须零错误）
- 改过界面 → `agent-browser` 截图给张总看

**门禁：** ✅ TypeScript 零错误（不改好不能提交）

### [3.4] 测试（Testing）

**目的：** 确保功能跟说好的完全一致。

**做法：**
- 复杂功能 → 挂 `browser-testing-with-devtools` 用真实浏览器测
- 有测试用例 → 挂 `qa` 或 `webapp-testing` 技能
- 我手动对照验收标准逐条检查
- 不同浏览器检查：Chrome / Firefox / Safari / Edge

**门禁：** ✅ 所有验收标准都过了

### [3.5] 安全审查

**目的：** 看看有没有安全漏洞。

**做法：**
- 涉及登录/密码/数据 → 挂 `security-and-hardening` 技能
- 跑 `pnpm audit` 检查第三方库有没有已知漏洞
- 检查：有没有高危漏洞、过期的依赖

**门禁：** ✅ 没有高危安全漏洞

### [3.6] 提交代码

**目的：** 把写好的代码存档到 Git。

**提交信息格式：**
```
<图标> <功能名称>

- 做了什么
- 为什么这么做
```

**图标对照表：**
| 图标 | 意思 | 例子 |
|------|------|------|
| 🎨 | UI 改动 | 调整按钮颜色、改布局 |
| ✨ | 新功能 | 加了登录功能 |
| 🐛 | Bug 修复 | 修了页面报错 |
| ♻️ | 重构 | 代码重写但功能不变 |
| 📝 | 文档 | 写 README |
| 🔧 | 配置 | 改环境变量 |
| ⚡ | 性能 | 加快页面加载 |
| 🧪 | 测试 | 加测试用例 |
| 🔒 | 安全 | 修安全漏洞 |
| 🚀 | 部署 | 发布上线 |
| ⏪ | 回滚 | 撤回到之前版本 |

---

## 张总只需要做的

1. 验收的时候看一下效果
2. 说"行了"或"这里还要改"
