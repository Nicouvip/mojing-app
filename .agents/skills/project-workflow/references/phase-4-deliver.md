# Phase 4：交付与发布

> **什么时候用：** 质量验证完了，要合并代码、发布上线。
> **通俗说：** 做好了，要正式拿出去用了。

---

## 做什么

```
Pre-flight检查 → 最终验证 → 打版本号 → 合并 → 构建发布 → 上线验证 → 回滚预案
```

---

## 步骤

### [4.1] 交付前完整性检查（Pre-flight）

**目的：** 最后一次全面检查，确保万无一失。

**做法：**
- 我挂 `verification-before-completion` 技能
- 逐项打勾：

**检查清单：**
- [ ] 设计方案里的需求全部做完了
- [ ] TypeScript 零错误
- [ ] 构建没有报错
- [ ] 界面跟设计稿对得上
- [ ] 手机/平板/电脑上显示都正常
- [ ] 数据为空时、加载中时、出错时都有显示（不是白屏）
- [ ] 键盘能操作、焦点不乱跳
- [ ] 搜索引擎能看到（meta标签齐全）

### [4.2] 最终验证

**目的：** 机器再完整跑一遍检查。

**做法：**
- `verify_changes full`（完整构建+类型检查+代码检查+测试）

**门禁：** ✅ 全部通过

### [4.3] 打版本号

**目的：** 每次发布都记一个版本号，出问题能知道是哪个版本。

**版本号规则（通俗版）：**
```
v主版本.次版本.补丁
```

| 例子 | 什么时候用 |
|------|-----------|
| v1.0.0 → v2.0.0 | 大改版，跟以前不兼容 |
| v1.0.0 → v1.1.0 | 加了新功能，不影响旧的 |
| v1.1.0 → v1.1.1 | 修了个 Bug，功能没变 |

**同时更新 CHANGELOG.md（更新记录）：**
```
## [v1.1.0] - 2026-07-08

### Added
- 新增了XX功能

### Changed
- 优化了XX

### Fixed
- 修复了XX Bug
```

### [4.4] 合并到主分支

**目的：** 功能代码正式合入主代码库。

**做法：**
- `git checkout master`
- `git merge feature/xxx`
- `git tag vX.Y.Z`（贴上版本标签）
- 删掉功能分支（已合并，留着没用）

### [4.5] 构建发布

**目的：** 生成正式版本并部署到服务器。

**做法：**
- 挂 `deploy-to-vercel` 或 `shipping-and-launch` 技能
- `pnpm build`（生成生产版本）
- 检查性能：
  - Lighthouse 评分 ≥ 90 分
  - 页面加载速度达标
  - 代码包大小没异常变大

**门禁：** ✅ 构建成功 + 性能达标

### [4.6] 上线验证

**目的：** 上线后确认真的能正常用。

**做法：**
- 用 `agent-browser` 打开线上地址
- 检查：页面能打开、核心功能正常、控制台没有报错
- 配置错误监控，以后出问题能发现

**门禁：** ✅ 线上正常 + 监控已开

### [4.7] 回滚预案（安全网）

**目的：** 万一上线出问题，知道怎么撤回来。

**回滚条件：** 上线后发现严重问题（比如用户用不了、数据出错）

**回滚方式：**
- 代码回到上一个版本 `git revert`
- 数据库也一起回退（如果有改动）
- 环境变量也回退

**回滚后要做的事：**
1. 确认服务恢复正常
2. 记录事故原因
3. 出修复方案
4. 修好了重新上线

### [4.8] 汇报

**做完后跟张总说：**
- 做了什么（一句话）
- 有什么要张总留意的
- 有没有遗留问题
- 下次从哪开始
