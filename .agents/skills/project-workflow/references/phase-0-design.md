# Phase 0：需求与设计

> **什么时候用：** 用户提出一个新想法、新功能、新需求时。
> **通俗说：** 把张总脑子里的想法，变成能动手做的方案。

---

## 做什么

```
张总的想法 → 确认理解 → 出方案 → 出设计稿 → 写文档 → 审查通过
```

---

## 步骤

### [0.1] 探索与澄清

**目的：** 确保我完全理解张总想要什么，不猜、不跑偏。

**做法：**
- 我按 AIM 框架回答（A角色 / I输入 / M任务）
- 如果需求不清楚 → 我一个一个问题问，不一次性问一堆
- 如果需要看竞品 → 我用 `agent-browser` 去调研同类产品
- 如果需要查资料 → 我用 `web_fetch` 拿信息

**门禁：** ✅ 张总说"对，就是这个意思"

### [0.2] 出技术方案

**目的：** 给出 2-3 种做法，让张总选。

**做法：**
- 涉及接口/API → 我自动挂 `api-and-interface-design` 技能
- 涉及数据结构 → 我自动挂 `domain-modeling` 技能
- 方案要写清楚：怎么架构、数据怎么流转、有什么风险
- 我会推荐一个最优方案并说明理由

**门禁：** ✅ 张总选了一个方案

### [0.3] UI/UX 设计（涉及界面时）

**目的：** 让张总先看到效果，再动手写代码。

**做法：**
- 我挂 `design-taste-frontend` 或 `high-end-visual-design` 技能
- 我做一个独立 HTML 设计稿（张总可以直接双击打开看）
- 派 `design-reviewer` 子代理审查设计是否合理
- 张总看了说"行"或"改"，直到满意

**门禁：** ✅ 张总说"这个设计可以"

### [0.4] 写设计文档

**目的：** 方案定下来后写下来，以后不会忘。

**存到：** `墨境/项目文档/序号-功能名称-设计文档.md`

**文档要写清楚：**
- 背景：为什么要做这个
- 方案：选了什么方案、为什么
- 怎么实现的：组件结构、数据怎么走、接口长什么样
- 怎么算做完：验收标准

**门禁：** ✅ 张总看了一遍说"没问题"

### [0.5] 设计评审（复杂功能）

**目的：** 在动手前再查一遍有没有漏洞。

**做法：**
- 派 `design-reviewer` 子代理只读审查设计
- 它看：设计有没有矛盾、有没有漏掉什么
- 发现问题我改，改完再给张总看

---

## 张总只需要做的

1. 说想法
2. 回答我的问题（我一个一个问）
3. 看方案，选一个
4. 看设计稿，说"行"或"改哪里"
5. 看设计文档，说"没问题"
