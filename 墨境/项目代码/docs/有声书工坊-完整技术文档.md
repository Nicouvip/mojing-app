```typescript
// 角色分析结果
interface CharacterAnalysis {
  name: string                    // 角色名
  gender: 'male' | 'female'       // 性别
  age: 'child' | 'young' | 'adult' | 'elderly'  // 年龄段
  personality: string              // 性格描述
  recommendedVoice: string         // 推荐音色ID
  recommendedEmotion: string       // 推荐情绪
}

// 段落分析结果
interface SegmentAnalysis {
  index: number                    // 段落序号
  type: 'narration' | 'dialogue'   // 类型：旁白/对话
  text: string                     // 文本内容
  characterName?: string           // 角色名（对话类型时）
  emotion: string                  // 情绪（平静/开心/悲伤/愤怒等）
  emotionIntensity: number         // 情绪强度 1-10
  recommendedVoice: string         // 推荐音色ID
  speed: 'slow' | 'normal' | 'fast'  // 语速
  needsPause: boolean              // 是否需要停顿
  pauseAfter: 'short' | 'normal' | 'long'  // 停顿时长
  specialNote?: string             // 特殊演播指导（气声、哽咽等）
}

// 旁白风格
interface NarrationStyle {
  overallTone: string              // 整体基调
  suggestedNarratorVoice: string   // 推荐旁白音色
  pacing: 'slow' | 'normal' | 'fast'  // 节奏
}

// 完整分析结果
interface AnalysisResult {
  characters: CharacterAnalysis[]
  segments: SegmentAnalysis[]
  narrationStyle: NarrationStyle
}
```

## 4. API 路由详解

### 4.1 POST /api/audiobook/analyze — AI 分析章节

**请求：** `{ text: string }`  
**响应：** `{ success, characters: CharacterAnalysis[], segments: SegmentAnalysis[], narrationStyle, meta }`

**引擎：** DeepSeek V4 Flash（直接 fetch，非引擎类）  
**环境变量：** `DEEPSEEK_API_URL` + `DEEPSEEK_API_KEY`

**流程：**
1. 文本超过 15000 字符会截断
2. 用 `buildAnalysisPrompt()` 构建提示词
3. 调用 DeepSeek API（超时 120s）
4. 提取 JSON（含 3 次重试）
5. 校验结构 → 返回

**提示词核心要求：**
- 识别每个段落是旁白还是对话
- 为每个角色推荐音色和情绪
- 为每个段落标注情绪强度(1-10)、语速、是否需要停顿
- 特殊处理：气声、哽咽、低语等填入 specialNote

---

### 4.2 POST /api/audiobook/generate — TTS 单段生成

**请求：**
```json
{
  "text": "要合成的文本",
  "voice": "冰糖",
  "emotion": "悲伤",
  "style": "低沉",
  "format": "wav",
  "emotionIntensity": 7,
  "speed": "slow",
  "specialNote": "气声，低声呢喃"
}
```
**响应：** `{ success, audio: base64, duration, format }`

**引擎：** `MiMoTTSEngine.generate()`

**参数传递链：**
```
前端 generateOne(seg)
  → POST /api/audiobook/generate
    → engine.generate({ text, voice, emotion, style, format, emotionIntensity, speed, specialNote })
      → 构建风格指导文本：styleParts = [emotion, "情感强度N/10", "语速缓慢/加快", specialNote, style]
      → messages: [{ role: 'user', content: styleParts.join('，') }, { role: 'assistant', content: text }]
      → MiMo API: mimo-v2.5-tts
```

---

### 4.3 POST /api/audiobook/import — 小说文本导入

**请求：** `{ projectId, text, title?, splitMode?: 'auto'|'manual'|'none' }`  
**响应：** `{ success, projectId, chapters: [{title, content}], stats }`

**分章逻辑：**
- `auto`：正则匹配「第X章」「第X节」「Chapter X」等标题 → 失败则按空行分
- `manual`：按空行分
- `none`：不分章

---

### 4.4 POST /api/audiobook/import-book — 画本导入（TXT/JSON）

**请求：** `{ format: 'txt'|'json', content: string, chapterId? }`  
**响应：** `{ success, title?, characters, segments, meta }`

**TXT 格式：**
```
## 角色表
林默：男，青年，清亮
苏晚：女，青年，温柔

## 画本内容
1 | 林默 | 平静 | 今天天气真好。
2 | 苏晚 | 温柔 | 是啊，阳光很暖。
```

**JSON 格式：**
```json
{
  "title": "画本标题",
  "characters": { "林默": { "gender": "male", "age": "young", "tone": "清亮" } },
  "lines": [
    { "index": 1, "type": "dialogue", "character": "林默", "emotion": "平静", "text": "今天天气真好。" }
  ]
}
```

### 4.5 POST /api/audiobook/import-docx — 画本导入（DOCX）

**请求：** `multipart/form-data { file: File(.docx) }`  
**响应：** `{ success, title, characters, segments, meta }`

**流程：**
1. 接收 FormData 二进制文件
2. `mammoth.extractRawText({ buffer })` 提取纯文本
3. 检测有声喵角色定义区（5个方括号格式）
4. 解析正文旁白/对话标记

**有声喵 DOCX 格式：**
```
【墨染青衣Nicou】【李教授】【男】【表面是受人景仰的教育楷模...】【10】
【cv嘉悦..】【女儿】【女】【长期遭受精神控制后觉醒...】【1】
【一束光的麦】【医生1】【男】【资深诊疗医生...】【2】
-------------------------------------------------------------
第1章 模范囚笼（一）
他对着话筒，声音温润如玉：【李教授-墨染青衣Nicou】"在我女儿大喜的日子里，我想分享真正的爱的教育……"
我接过话筒，指尖冰凉：【女儿-cv嘉悦..】"爸爸，不如让我们分享一些更真实的教育。"
```

**解析规则：**
- `【cv名】【角色名】【性别】【角色描述】【出场数】` → 角色定义
- `【角色名-cv名】"对话内容"` → 标记自动剔除，只保留对话文本配音
- 标记前的文本 → 旁白
- 无标记的纯文本段落 → 整段作为旁白

---

### 4.6 POST /api/audiobook/merge — 合并导出

**请求：**
```json
{
  "segments": [{ "audioBase64": "...", "duration": 3.5 }],
  "format": "wav"  // wav | mp3 | m4b
}
```
**响应：** `{ success, audio: base64, duration, format, mergedSegments }`

**流程：**
1. 解码所有 base64 → 提取裸 PCM
2. 在段间插入静音（不同角色 0.5s / 同角色 0.25s）
3. 拼接所有 PCM
4. 添加 WAV 头（24kHz, 16bit, mono）
5. 若 mp3/m4b → 调用 ffmpeg 转码

## 5. 引擎层详解

### 5.1 MiMoTTSEngine（mimo-tts.ts）

**API 地址：** `https://token-plan-cn.xiaomimimo.com/v1/chat/completions`  
**模型：** `mimo-v2.5-tts`  
**认证：** `api-key` header（`MIMO_API_KEY` 环境变量）

**方法：**
- `generate(params)` → 同步生成，返回 `{ audioBuffer, duration, format }`
- `generateStream(params)` → `AsyncGenerator<Buffer>`，流式生成
- `getVoices()` → 返回预置音色列表

**风格指导构建逻辑：**
```typescript
const styleParts: string[] = []
if (params.emotion) styleParts.push(params.emotion)
if (params.emotionIntensity && params.emotionIntensity > 5) styleParts.push(`情感强度${params.emotionIntensity}/10`)
if (params.speed === 'slow') styleParts.push('语速缓慢')
if (params.speed === 'fast') styleParts.push('语速加快')
if (params.specialNote) styleParts.push(params.specialNote)
if (params.style) styleParts.push(params.style)

// messages 结构
messages = [
  { role: 'user', content: styleParts.join('，') },      // 风格指导
  { role: 'assistant', content: params.text }              // 要合成的文本
]
```

**重试机制：** `fetchWithRetry(url, options, timeoutMs=90000, maxRetries=2)`

---

### 5.2 MiMoASREngine（mimo-asr.ts）

**API 地址：** 同 TTS  
**模型：** `mimo-v2.5-asr`

**方法：**
- `transcribe(params)` → 识别单个音频
- `transcribeBatch(params)` → 批量识别
- `generateLRC(timestamps)` → 生成 LRC 字幕
- `generateSRT(timestamps)` → 生成 SRT 字幕

---

### 5.3 MiMoVoiceCloneEngine（mimo-voice-clone.ts）

**模型：** `mimo-v2.5-tts-voiceclone`

**方法：**
- `clone(params)` → 克隆并生成音频
- `validateSample(buffer, mimeType)` → 校验样本（≤7.5MB, mp3/wav）
- `getSampleAdvice()` → 返回录制建议

**特殊：** 音频样本放在 `audio.voice` 字段（DataURL），文本放在 messages

---

### 5.4 MiMoVoiceDesignEngine（mimo-voice-design.ts）

**模型：** `mimo-v2.5-tts-voicedesign`

**方法：**
- `design(params)` → 设计音色并生成音频
- `preview(description, previewText?)` → 预览音色
- `getDesignSuggestions(characterType)` → 按角色类型给出描述建议

---

### 5.5 DOCX 解析器（docx-parser.ts）

**依赖：** `mammoth`（npm 包）

**方法：**
- `parseDocxBuffer(buffer)` → 用 mammoth 提取文本后解析
- `parseDocxText(plainText)` → 直接从纯文本解析

**有声喵格式解析流程：**
1. 检测角色定义区（连续5个方括号的行）
2. 跳过分隔线 `----`
3. 跳过章节标题 `第X章`
4. 正文：用正则 `/【([^】]+)-([^】]+)】/g` 匹配对话标记
5. 标记前文本 → 旁白段落
6. 标记后引号内文本 → 对话段落
7. 纯文本行 → 整行旁白

---

### 5.6 段落合并工具（merge-segments.ts）

**核心函数：**
- `mergeConsecutiveSegments(segments, characters)` → 合并连续旁白（≤300字）和同角色对话（≤200字）
- `splitDialoguePrefix(segments)` → 从对话中分离"XXX说："前缀为旁白
- `processAnalysisResult(segments, characters)` → 完整管道（前缀分离 → 连续合并 → 重编号）

---

### 5.7 字幕生成器（srt-generator.ts）

**方法：**
- `generateSRT(timestamps)` → 标准 SRT 格式
- `generateLRC(timestamps)` → LRC 格式
- `generateFakeTimestamps(segments, durationSec)` → 按字符数均匀分配时间戳（无 ASR 时使用）

## 8. 数据持久化方案

### 8.1 三层存储架构

| 层 | 技术 | 用途 |
|----|------|------|
| **内存** | React state | 运行时音频缓存、UI 状态 |
| **localStorage** | 浏览器本地存储 | 分析结果缓存（`mojing_analysis_*`） |
| **Turso (libSQL)** | 云端数据库 | 项目/章节/角色档案/大纲/伏笔等持久化数据 |

### 8.2 Turso 表结构

10 张表：`projects, chapters, volumes, character_profiles, outlines, world_settings, foreshadows, cooling_states, writing_plans, chapter_reports`

**⚠️ 关键发现：有声书数据不持久化**

有声书的核心数据（`AudiobookProject`, `AudioChapter`, `AudioSegment`, `VoiceProfile`）在 `src/lib/audiobook/index.ts` 中**只定义了接口，没有数据库表和 store 层实现**。所有有声书数据：
- 分析结果：纯 API 返回，前端内存持有 + localStorage 缓存
- 音频：base64 通过 API 返回，无持久化
- 画本：纯 API 解析返回

**后果：** 刷新页面后音频丢失，需要重新生成。

## 附录D：关键设计决策记录

### D1. 为什么用 MiMo 而不是其他 TTS？

MiMo V2.5 支持：
- **8 款预置音色**（中英文）
- **精细情绪控制**（9种基础 + 8种复合 + 5种语气 + 4种音质 + 方言 + 角色）
- **VoiceClone**（音频样本克隆）
- **VoiceDesign**（自然语言描述生成音色）
- **ASR + 时间戳**（同步字幕生成）
- **低延迟**（适合实时生成）

### D2. 为什么用 DeepSeek V4 Flash 做分析？

- **速度快**（Flash 模型，30-60秒分析一个章节）
- **中文能力强**（角色识别、情绪分析、音色推荐）
- **成本低**（比 GPT-4 便宜 10 倍+）
- **JSON 输出稳定**（配合规则型 Prompt）

### D3. 为什么有声书数据不持久化？

**历史原因：** 有声书功能是从「文本分析工具」演进而来，最初只是分析+预览，不涉及音频存储。随着功能扩展（TTS生成、合并导出），音频一直以 base64 在前端内存中传递，没有设计数据库表。

**后果：** 刷新页面后所有生成的音频丢失，需要重新生成。

**解决方案：** 新增 `audio_segments` 表（详见路线图 P0）。
